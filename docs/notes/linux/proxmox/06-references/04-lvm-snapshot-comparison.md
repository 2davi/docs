# LVM-Thick / LVM-Thin / ZFS 스냅샷 메커니즘 비교: CoW vs RoW 개념 정립
---
## 발단: 질문의 출발점
"LVM-Thick과 LVM-Thin 둘 다 CoW 방식 아니냐? 변경된 블록만 추적하고, 스냅샷 시 변경된 블록만큼만 추가 저장하는 것도 동일하지 않냐?"는 질문에서 논의가 시작되었다.

결론부터 말하면, 이 질문의 직관은 **부분적으로 맞고 부분적으로 틀렸다.** LVM-Thick과 LVM-Thin은 "원본을 덮어쓰지 않는다"는 공통점이 있지만, 실제 동작 방식은 명확히 다르다. 그리고 ZFS와 LVM-Thin은 결과적으로 거의 동일한 메커니즘을 사용한다.

***
## 1. LVM-Thick 스냅샷: 진짜 CoW
LVM-Thick 스냅샷은 교과서적 의미의 **CoW(Copy-on-Write)** 를 사용한다.

**동작 순서:**

1. 스냅샷을 찍는 순간, 원본 LV의 메타데이터(포인터)만 복사해서 스냅샷 LV를 만든다.
2. 이후 원본 LV에 **쓰기가 발생하면**, 변경 전 블록을 **스냅샷 전용 공간에 먼저 복사**한다.
3. 복사가 완료된 다음, 원본 LV의 해당 블록을 새 데이터로 덮어쓴다.

**핵심:** "원본 블록 → 스냅샷 공간으로 복사"라는 **명시적인 복사 동작**이 쓰기마다 발생한다.

**제약:**
- 스냅샷 전용 공간을 미리 고정 크기로 확보해야 한다.
- 그 공간이 꽉 차면 스냅샷이 invalid 상태가 된다.
- 쓰기마다 복사 오버헤드가 발생해 성능 부담이 있다.

***
## 2. LVM-Thin 스냅샷: RoW (Redirect-on-Write)
LVM-Thin은 **RoW(Redirect-on-Write)** 방식을 사용한다.

**동작 순서:**

1. 스냅샷을 찍으면, 원본 Thin Volume과 스냅샷이 **같은 Thin Pool 블록 공간을 공유**한다.
2. 이후 원본에 쓰기가 발생하면, 원본 블록을 복사하지 않고 **새로운 블록 위치에 바로 데이터를 기록**한다.
3. 원본 Thin Volume의 해당 블록 포인터를 새 위치로 교체한다.
4. 스냅샷은 **기존 블록 포인터를 그대로 유지**한다 — 이전 데이터를 자동으로 보존하는 효과.

**핵심:** 블록 복사 동작이 없다. 포인터 리다이렉트만으로 스냅샷이 이전 데이터를 참조하도록 유지한다.

**장점:**
- 쓰기 오버헤드가 LVM-Thick보다 작다 (복사 없음).
- Thin Pool 공간을 유동적으로 공유해 효율이 높다.
- Proxmox의 VM 스냅샷 및 Linked Clone이 이 구조를 활용한다.

***
## 3. ZFS: CoW라고 불리지만 동작은 RoW에 가깝다
ZFS는 공식적으로 **CoW 파일시스템**으로 분류된다. 그러나 실제 동작을 보면:

1. 기존 블록을 덮어쓰지 않는다.
2. 새 데이터를 **완전히 새로운 위치(zpool 내 다른 블록)에 먼저 기록**한다.
3. 메타데이터(포인터, uberblock)를 새 위치로 교체한다.
4. 기존 블록은 더 이상 참조되지 않으면 해제된다.

이 과정에서 **블록 복사 동작이 없다.** "새 위치에 쓰고 포인터를 바꾼다"는 점에서 LVM-Thin의 RoW와 결과적으로 동일하다.

ZFS를 CoW라고 부르는 것은 역사적 관습과 "원본을 보존한다"는 넓은 의미의 사용이다. 엄밀한 메커니즘으로 보면 ZFS도 RoW에 해당한다는 시각이 있으며, 이는 타당하다.

***
## 4. 세 방식의 핵심 비교
| 항목 | LVM-Thick (CoW) | LVM-Thin (RoW) | ZFS (CoW로 불림) |
|---|---|---|---|
| 원본 블록 덮어쓰기 | X | X | X |
| 명시적 블록 복사 발생 | **O** (쓰기마다) | X | X |
| 새 위치에 데이터 기록 | X (원본 위치 유지) | O | O |
| 포인터 교체 | O | O | O |
| 스냅샷 공간 | 고정 크기 별도 확보 필요 | Thin Pool 유동 공유 | zpool 내 유동 관리 |
| 쓰기 오버헤드 | 높음 (복사 비용) | 낮음 | 낮음 |

***
## 5. "CoW vs RoW" 구분의 실무적 의미
이 두 용어를 칼같이 구분해야 하는 상황은 **스토리지 시스템 구현 레벨**에서의 이야기다. 실무 운영과 Proxmox 사용 맥락에서는 다음 두 가지만 구분하면 충분하다.

- **진짜 CoW (LVM-Thick 스냅샷)**: 쓰기마다 블록 복사가 발생한다. 스냅샷 공간 관리가 필요하고 성능 부담이 있다.
- **RoW 계열 (LVM-Thin, ZFS)**: 복사 없이 새 위치에 쓰고 포인터만 교체한다. 성능 효율이 높고 스냅샷이 가볍다.

ZFS와 LVM-Thin을 같은 계열로 묶어서 이해하는 것이 실무적으로 더 유용하다.
